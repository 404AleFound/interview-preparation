# 可重复读和幻读


## 幻读回顾
当同一个查询在不同的时间产生不同的结果集时，事务中就会出现所谓的幻象问题。例如，如果 select 执行了两次，但是第二次返回了第一次没有返回的行，则该行是“幻象”行。

## 快照读如何避免幻读

## 当前读如何避免幻读
`mysql` 中除了普通查询是快照读，其他都是当前读，比如 `update`、`insert`、`delete`，这些语句执行前都会查询最新版本的数据，然后再做进一步的操作。

另外，`select ... for update` 这种查询语句是当前读，每次执行的时候都是读取最新的数据。

对于数据表：
|id|name|score|
|---|---|---|
|1|Tom|50|
|1|Frank|50|
|3|Jack|50|
|4|Anna|60|

如果当前读不加锁，会出现如下幻读：
|事务A|事务B|
|---|---|
|**`begin;`**||
|**`select name from t_stu where id > 2;`**||
|`--- results: Jack, Anna ---`||
||**`begin;`**|
||**`insert into t_stu values(5, "John, 100");`**|
||**`commit;`**|
|**`select name from t_stu where id > 2;`**||
|`--- results: Jack, Anna, John ---`||
|**`commit;`**||

因此，InnoDB 引擎未来解决可重复读隔离级别下，使用当前都造成的幻读问题，提出了间隙锁：
- 对上述操作，在当前读时加入 id 范围为 `(2, +∞)`的 next-key lock。
- 然后，事务 B 在执行插入语句的时候，判断插入的位置被事务 A 加了 next-key lock ，于是事务 B 会生成一个插入意向锁，同时进入等待状态，直到事务 A 提交了事务。这就避免了事务 B 插入新记录而导致事务 A 发生幻读的现象。

|事务A|事务B|
|---|---|
|**`begin;`** ||
|**`select name from t_st where id > 2 for update;`**||
|`--- results: Jack, Anna ---`||
|`--- results: lock id (2, +∞) ---`||
||**`begin;`**|
||**`insert into t_st values(5, "John, 100");`**|
||`--- results: waitting... ---`|
|**`select name from t_st where id > 2 for update;`**|`--- results: waitting... ---`|
|`--- results: Jack, Anna ---`|`--- results: waitting... ---`|
|**`commit;`**|`--- results: waitting... ---`|
||`--- results: insert successfully! ---`|
||**`commit;`**|


### 伏笔
可以看到上面是在可重复读隔离级别中避免幻读现象的手段，但是也可以看出其中还是有不少隐患：
- 对于快照读，通过 MVCC 版本控制避免幻读，每条数据虽然可以通过 隐藏列和 Read View 对事物进行接受与拒绝，但是其对创造自己的事物总是会留一个后门。
- 对于当前读，可以看到开启 next-key lock 的时机对于是否会出现幻读现象是很关键的。

## 幻读发生的情况
可重复读隔离级别下虽然很大程度上避免了幻读，但是还没有能完全解决幻读。

同样，对于数据表：
|id|name|score|
|---|---|---|
|1|Tom|50|
|1|Frank|50|
|3|Jack|50|
|4|Anna|60|

### 快照读发生幻读

|事务A|事务B|
|---|---|
|`begin;`                                               |                       |
|`select name from t_st where id = 5;`||
|`--- results: None ---`||
||`begin;`|
||`insert into t_st values(5, "John, 100");`|
||`commit;`|
|`update t_st set name = 'Hellen' where id = 5;`|                                            |
| `select name from t_st where id = 5;`           |                                            |
| `--- results: Hellen ---`                       |                                            |

在可重复读隔离级别下，事务 A 第一次执行普通的快照读生成一个 Read View，之后事务 B 向 表中新插入了一条 id = 5 的记录并提交。接着，事务 A 对 id = 5 的记录进行了更新操作，在这个时候，这条新记录的 `trx_idx` 就编程了事务 A 的事务 id，这条记录就对事务 A 可见了，于是发生了幻读。

### 当前读发生幻读

除此之外，下面这个场景也会发生幻读：

- T1 时刻，事务 A 先执行快照读语句：`select * from t_st where id > 10` 得到 3 条记录。
- T2 时刻，事务 B 往数据表中插入一个 id = 200 的记录并提交;
- T3 时刻，事务 A 再执行当前读语句：`select * from t_st where id > 100 for update` 便马上会得到 4 条记录，此时也发生了幻读现象。

避免上述情况下发生幻读的解决措施，就是在开启事务后，尽可能马上执行 `select ... for update`这类当前读的语句，因为其会对表格中记录加上 next-key lock，从而避免其他事务插入一条新的记录。
