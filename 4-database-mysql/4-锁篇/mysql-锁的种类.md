## 全局锁
### 全局锁的使用
```sql
flush tables with read lock
```
上述语句执行后，整个**数据库**就处于只读状态了，此时其他线程执行下述操作都会被阻塞：
- 对数据的增删改操作，比如 `insert`、`delete`、`update` 等语句。
- 对表结构的更改操作，比如 `alter table`、`drop table` 等语句。
如果要释放全局锁，使用下述语句：
```sql
unlock tables
```
### 全局锁的应用场景
全局锁的主要用途是**全库逻辑备份**，这样在备份期间，不会因为数据或表结构的更新，而出现备份的文件数据和预期的不一致。

### 全局锁的缺点
由于加上全局锁，意味着整个数据库都是只读状态。那么如果数据库里有很多数据，备份就会花费很多时间，关键是备份期间，业务只能读，而不能更改数据，这样会导致业务的停滞。

### 全局锁的改进
为了规避上述不足，对于支持可重复读隔离级别的数据库，那么在备份数据库之前先开启事务，会先创建 Read View ，然后整个事务执行期间都在用这个 Read View，而且由于 MVCC 的支持，备份期间业务依然可以对数据进行更新。
## 表级锁
### 表锁
使用下列语句对表进行加锁：
```sql
// 表级别的共享锁，也就是读锁。
// 允许当前会话读取被锁定的表，但阻止其他会话对这些表进行写操作。
lock tables t_student read;
// 表级别的独占锁，也就是写锁。
// 允许当前会话对表进行读写操作，但阻止其他会话对这些表进行任何操作。
lock tables t_student write;
```

要释放表锁，可以使用下面这条命令，会释放当前会话的所有表锁：
```sql
unlock tables
```
另外，当会话退出后，也会释放所有表锁。

在还没有出现细粒度的锁的时候，表锁是最常用的处理并发的方式，不过尽量避免在 InnoDB 引擎的表使用表锁，因为表锁的颗粒度太大，会影响并发性能，InnoDB 独特之处就在于实现了颗粒度更细的行级锁。
### 元数据锁
**元数据锁的目的**
元数据锁是为了保证当用户对表执行 CRUD 操作时，防止其他线程对这个表结构做了变更。

当有线程在执行 select 语句的期间，如果有其他线程要更改该表的结构，那么将会被阻塞，直到执行完 select 语句。

反之，当有线程对表结构进行变更的期间，如果有其他线程执行了 CRUD 操作，那么就会被阻塞，直到表结构变更完成。

**元数据锁的使用与释放**
元数据锁的使用，不需要显式地调用，因为当我们对数据库进行操作时，会自动给这个表加上 MDL：
- 对一张表进行 CRUD 操作时，加的是 MDL 读锁
- 对一张表进行结构变更操作时，加的是 MDL 写锁

MDL 是在事务提交之后才会释放的，这意味着事务执行期间，MDL 是一直持有的。

**元数据锁的阻塞问题**
当数据库有一个长事务，那么在对表结构做变更操作的时候，可能会发生意向不到的事情：
- 线程 A 启用事务，但是一直不提交，然后执行一条 select 语句，此时表就被加上了 MDL 读锁
- 然后，线程 B 也执行了同样的操作，此时并不会阻塞，因为读读不冲突
- 接着，线程 C 想修改表字段，需要申请一个 MDL 写锁，但是由于事务 A 并没有提交，且由于读写互斥，线程 C 就会被阻塞。

由于线程申请 MDL 锁的操作会形成一个队列，队列中写锁的优先级高于读锁，一旦出现 MDL 写锁等待，会阻塞后续该表的所有 CRUD 操作。

注意，MDL 写锁不会插队，但它会阻塞后续对该表的所有操作。具体表现为：当有线程申请 MDL 写锁（如表结构变更）时，只有在当前持有的所有 MDL 读锁释放后，写锁才能获得；而在写锁等待期间，新的读锁和写锁请求都会被阻塞，形成排队等待。因此，写锁不会插队，但会导致后续操作全部等待写锁完成。

### 意向锁
**意向锁的作用**
添加意向锁的目的是为了快速判断表里是否有记录被加锁。

如果没有意向锁，那么加独占表锁时，就需要遍历表里所有记录，查看是否有记录存在独占锁，这样效率会很慢。

而意向锁可以在对记录加独占锁之前，先加上表级别的意向独占锁，那么在加独占表锁的时候，直接查看该表是否有意向独占锁，如果有就意味着表里已经有记录被加了独占锁，这样就不用去遍历表里的记录。

**意向锁的使用**
注意：
- 在使用 InnoDB 引擎的表里对某些记录加上共享锁之前，需要现在表级别加上一个意向共享锁。
- 在使用 InnoDB 引擎的表里某些记录加上独占锁之前，需要现在标记别加上一个意向独占锁。

也就是，当执行插入、更新、删除操作，需要先对表加上意向独占锁，然后对该记录加独占锁。

而普通的 select 是不会加行级锁的，因为普通的 select 语句是利用 MVCC 实现一致性读，是无锁的。

不过 select 也是可以对记录加共享锁和独占锁的，具体方式如下：
```sql
// 先在表上加上意向共享锁，之后对读取的记录加共享锁
select ... lock in share mode;
// 先在表上加上意向独占锁，然后对读取的记录加独占锁
select ... for update;
```

**意向共享锁和意向独占锁是表级锁，是不会和行级锁冲突的；而且意向锁之间也不会发生冲突，只会和共享表锁和独占表锁发生冲突。**

### AUTO-INC 锁
**AUTO-INC 锁的作用**
表里的主键通常会被设置为自增的，这是通过对主键字段声明 `AUTO_INCREMENT`实现的。之后插入数据时，可以不指定主键的值，数据库会主动给主键赋值递增的值，这主要是通过 AUTO-INC 锁实现的。

AUTO-INC 锁是特殊的表锁机制，不是在一个事务提交后才释放的，而是再执行外插入语句后立刻释放的。

**AUTO-INC 锁的使用**
在插入数据时，会加一个表级别的 AUTO-INC 锁，然后为被 AUTO_INCREMENT 修饰的字段赋值递增的值，等插入语句执行完成后，才会把 AUTO-INC 锁释放掉。

那么，一个事务在持有 AUTO-INC 锁的过程中，其他事务如果要想该表插入语句都会被阻塞，从而保证插入数据时，被AUTO-INCREMENT 修饰的字段的值是连续递增的。

**AUTO-INC 锁的轻量化**
上述操作在大量数据插入时，会影响插入性能，因为需要等待插入语句完成后才释放掉锁。

为此 InnoDB 引擎提供了一种更加轻量级的锁。其在插入数据时，会为被 AUTO_INCREMENT 修饰的字段加上轻量级锁，然后给该字段赋一个递增的值，之后立刻释放掉这个锁，不用等到整个插入语句执行完后才释放。

注意，轻量级的锁（如 InnoDB 针对 AUTO_INCREMENT 字段的轻量级锁）本质上仍然是表级锁，而不是行级锁。它的作用范围是整个表，但持锁时间更短，仅在分配自增值时加锁，分配完成后立即释放，从而提升并发性能。

## 行级锁
InnoDB 引擎是支持行级锁的，而 MyISAM 引擎并不支持行级锁
普通的 select 语句是不会对记录加锁的，因为它属于快照读。如果要在查询时对记录加行级锁，可以使用下面这两个方式，这种查询加锁的语句成为锁定读：
```sql
// 对读取的记录加共享锁
select ... lock in share mode;
// 对读取的记录加独占锁
select ... for update;
```
共享锁（S锁）满足读读共享、读写互斥，独占锁（X锁）满足写写互斥、读写互斥。
### Record Lock
Record Lock 称为记录锁，锁住的是一条记录。而且记录锁也有 S 锁和 X 锁之分的：
- 当一个事务对一条记录加 S 型记录锁后，其他事务也可以继续对该记录加 S 型记录所，但是不可以对该记录加 X 型锁。
- 当一个事务对一条记录加 X 型记录锁后，其他事务既不可以对该记录加 S 型记录锁，也不可以对该记录加 X 型记录锁。

当事务执行 `commit` 后，事务过程中生成的所有锁都会被释放。
### Gap Lock
Gap Lock 称为间隙锁，存在于可重复读隔离级别和串行化隔离级别，目的是为了解决可重复读隔离级别下幻读的问题。

假设，表中有一个范围 id 为 `(3,5)` 的间隙锁，那么其他事务就无法插入 `id = 4` 这条记录了，这样就有效的防止了幻读现象的发生。

间隙锁虽然存在 X 型间隙锁和 S 型间隙锁，但是并没有区别，间隙锁之间是兼容的，即两个事务可以同时持有包含共同间隙范围的间隙锁，并不存在互斥关系，因为间隙锁的目的是防止插入幻影记录而提出的。
### Next-key Lock
Next-Key Lock 称为临键锁，是 Record Lock 和 Gap Lock 的组合，锁定一个范围，并且锁定记录本身。

假设，表中有一个范围 id 为 `(3, 4]` 的 next-key lock，那么其他事务就不能插入 `id = 4` 的记录，也不能修改 `id = 5` 这条记录。

所以，next-key lock 既能保护该记录，又能阻止其他事务将新记录插入到被保护记录前面的间隙中。

由于 next-key lock 是包含间隙锁和记录锁的，如果一个事务获取了 X 型的 next-key lock，那么另外一个事务在获取相同范围的 X 型的 next-key lock 时，是会被阻塞的。

虽然相同范围的间隙锁是相互兼容的，但对于记录锁，我们要考虑 X 型与 S 型关系。
### 插入意向锁
一个事务在插入一条记录的时候，需要判断插入位置是否已被其他事务加了间隙锁，如果有的化，插入操作会被阻塞，直到拥有间隙锁的那个事务提交位置。

在此期间会生成一个插入意向锁，表明有事务想在某个区间插入新记录，但是现在处于等待状态。

插入意向锁时行级别的锁，是一种特殊的间隙锁，并不是意向锁。

两个事务不能在同一时间，一个拥有间隙锁，另一个拥有该间隙锁区间内的插入意向锁。

**插入意向锁的释放时机**

插入意向锁是在事务执行插入操作并因间隙锁阻塞时自动加上的。当阻塞解除（即持有间隙锁的事务提交或回滚）并且插入操作完成后，插入意向锁会立即被释放。此外，如果事务主动回滚或提交，所有持有的插入意向锁也会随之释放。插入意向锁的生命周期通常很短，仅在等待插入期间存在。
