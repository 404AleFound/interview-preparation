## 隔离级别的概念
|隔离级别   |概念|
|---|---|
|读未提交   |指一个事务还没提交时，它做的变更就能被其他事务看到|
|读提交     |指一个事务提交后，它做的变更才能被其他事务看到|
|可重复读   |指一个事务执行过程中看到的数据，一直跟这个事务启动时看到的数据是一致的|
|串行化     |指会对记录加上读写锁，在多个事务对这条记录进行读写操作时，如果发生了读写冲突，后访问的事务必须等前一个事务执行完成，才能继续执行|


## 隔离级别的水平
隔离级别水平高低排序如下：
```
读未提交 < 读提交 < 可重复读 < 串行化
```

针对不同级别，并发事务时可能发生的现象也不同，如下：

|隔离级别|可能发生的事故|
|---|---|
|读未提交   |脏读、不可重复读、幻读|
|读提交     |不可重复读、幻读|
|可重复读   |幻读|
|串行化     |无|

注意，mysql 在可重复读隔离级别下，可以很大程度避免幻读现象的发生，所以 mysql 并不会使用串行化隔离级别来避免幻读现象的发生，因为使用串行化会影响 mysql 的性能。

mysql 针对快照读和当前读，分别通过 MVCC 和 锁的机制实现了可重复读：
- 快照读（普通 select 语句），通过 MVCC 当时解决了幻读，因为在可重复读隔离级别下，事务执行过程中看到的数据，一直跟这个事务启动时看到的数据是一致的，即使中途有其他事务插入了一条数据，也是无法查询出来的。
- 当前读（select ... for update 语句），通过 next-key lock 方式解决的幻读，因为当执行 select ... for update 语句的时候，会加上 next-key lock，如果有其他事务在 next-key lock 范围内插入一条记录，那么这个插入语句就会被阻塞，无法成功插入，这样就很好避免了幻读问题。

## 隔离级别的实现

- 对于读未提交，因为可以读取到未提交的数据，所以直接读取最新的数据就好了
- 对于串行化隔离级别，通过加读写锁的方式来避免并行访问
- 对于读提交和可重复读隔离级别的事务来说，是通过 Read View 来实现的，它们的区别在于创建 Read View 的时机不同。读提交隔离级别是在每个语句执行前创建 Read View，可重复读隔离级别是在每个事务启动时，创建一个 Read View，然后整个事务期间都使用这个 Read View。

注意，执行开始事务命令，并不意味着启动了事务，在 mysql 有两种开启事务的命令，分别是：
- `begin/start transaction` 命令
- `start transaction with consistent snapshot` 命令

这两种开启事务的命令，事务的启动时机是不同的：
- 执行了 `begin/start transaction` 命令后，并不代表事务启动了，在执行这个命令后，执行第一条 select 语句，才是事务真正启动的时机
- 执行 `start transaction with consistent snapshow` 命令，会马上启动事务
