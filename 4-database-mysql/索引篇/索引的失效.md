有些情况下，程序员撰写的语句可能会导致索引失效，导致 mysql 服务器走了全表扫描，最后虽然得到了索引结果，但是查询的性能大打折扣。

## 情况1：对索引使用左或者左右模糊匹配
### 什么是左或者左右模糊匹配？
也就是使用 `like %xx` 或者 `like %xx%` 这两种方式都会造成索引失效。

### 左或者左右模糊匹配索引失效原因？
因为索引 B+Tree 是按照索引值有序排泄存储的，只能根据前缀进行比较。

假设我们要查询 name 字段前缀为 “林” 的数据，也就是 `name link '林%'`，扫描索引的时候，会根据林这个字的拼音大小进行检索查询。

## 情况2：对索引使用函数
如下语句，对索引使用了函数：
```sql
select * from t_user where length(name)=6;
```

这样的话，因为索引维护的是索引原始字段的有序性，经过函数计算后的值，无法保证其有序性，也就无法走索引了。

**但是**，从 MySQL8.0 开始，索引特性增加了函数索引，即可以针对函数计算后的值建立一个索引，也就是该索引的值是函数计算后的值，所以就可以通过扫描索引来查询数据了。

```sql
alter table t_user add key idx_name_length ((length(name)));
```
## 情况3：对索引进行表达式计算
如下语句，对索引进行了表达式计算：
```sql
select * from t_user where id + 1 = 10;
```
这使得索引失效的原因和使用函数的类似，表达式计算后的值不是索引的原始值，只能通过把索引字段的取值都取出来，然后一次进行表达式的计算来进行条件判断，因此采用的是全表扫描的方式。

**但是**，如下类似的语句是可以走索引的：
```sql
select * from t_user where id = 10 - 1;
```

因为，这并不是在索引字段进行表达式计算了。

## 情况4：对索引隐式类型转换
如果索引字段是字符串类型，但是在条件查询中，输入参数是整型的话，索引也会失效。

但是，如果索引字段是整型类型，查询条件中是输入参数即使为字符串，也不会导致索引失效，还可以走索引扫描。
### mysql 数据类型转换规则
在 mysql 中在遇到字符串和数字比较的时候，会自动把字符串转换为数字，然后再进行比较。

### 隐式类型转换下的索引失效
**对于：**
```sql
select * from t_user where phone = 13000000001;
// type of index phone is char
```
在这里，mysql 会将索引自动转换为数字，因此上述语句相当于：
```sql
select * from t_user where CAST(phone AS signed int) = 13000000001;
```
也就是相当于对 phone 字段使用了函数，也就不能走索引了。
**对于：**
```sql
select * from t_user where id = "1";
// type of index id is int
```
进行隐式转换后，相当于：
```sql
select * from t_user where id = CAST("1" AS signed int);
```
可以看到，索引字段并没有用任何函数，CAST 函数是用在了输入参数，因此是可以走索引扫描的。
## 情况5：联合索引非最左匹配
对主键字段建立的索引叫做聚簇索引，对普通字段建立的索引叫做二级索引。那么多个普通字段组合在一起创建的索引就叫做联合索引，也叫组合索引。

联合索引要能正确使用需要遵循最左匹配原则，也就是按照最左优先的方式进行索引的匹配。
## 情况6：`WHERE` 子句中的 `OR`
在 WHERE 子句中，如果在 OR 前的条件列是索引列，而在 OR 后的条件列不是索引列，那么索引会失效。
```sql
select * from t_user where id = 1 or age = 18;
// id is index column, while age is not
```
这是因为 OR 的含义就是两个只要满足一个即可，因此只有一个条件列是索引列是没有意义的，只要有条件列不是索引类，就会进行全表扫描。

解决方法就是将非索引类设置为索引列。
